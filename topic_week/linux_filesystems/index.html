Nous 2022-03-17 18:51
I remember when the rule of thumb for a home system was 1.5 times the memory for a swap size.
That is an old rule when 256MB-2GB RAM was common. These days if you have 32GB then even 32GB of swap can cause issues when you have a program that has a memory leak. Too much ram delays the OOM killer and can cause your system to lock up with thrashing your system. Unless you need to hybernate (at which point your swap needs to be as big as the total amount of memory all your application can ever use).
I would say start with 1-2GB, and if you find you never use all of it then that is enough, if you find you fill it up often then you might want a bit more. If you go for large amounts of swap though you will want to look into the early oom killers to stop runaway processes.
Note that not having enough swap can also cause your system to lock up thrashing the fillesystem cache under heavy load.
unixpgmr 2022-03-17 18:58
Not really about swap, but I was working at a company that did programming on IBM AS/400 and one of the programmers accidently created an infinite loop that wrote to disk. He didn't realize it until everybody else was seeing significant slow down of their work. So he went to the console to kill it and the console was taking over 5 minutes to actually respond to each command. So we shut the machine down - hard. If you have never shut down down hard, it is a terrible process. The AS/400 filesystem is really a relational database with indexes loaded in memory. When you shutdown the system, if there are any indexes not written to disk, they are update on disk. When you shut an AS/400 down hard, you essentially pull the plug (figuratively). When the system comes up, it knows it was shut down hard, and reads the entire filesystem rebuilding those indexes. If you have lots of file and lots of big file, that can take hours.
