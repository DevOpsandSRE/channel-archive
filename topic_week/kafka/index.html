Hmmm. You might be less confused by just looking it up, but I'll give it an attempt: A message queue is basically exactly what it sounds like, it's a queue or series of queues that hold messages that can usually be arbitrary byte data. It's very useful for something like when it comes to scale and distribution. For example if I have a part of my app that's scaled down, put a whole bunch of messages into the queue one for each notification email that should be sent to a user let's say, I could have a hundred different workers that are all pulling from the queue so that I get a better balanced workload. Also most message queues have some form of ack/nack system, which allows you to guarantee that of messages delivered at least once, so that if one of your workers crashes or gets turned off in the middle of trying to process the message, you can retry easily

Nous 2021-08-02 14:52
I would argue that Kafka is not a message queue, it is more of an event system. The difference being that message queues do not hold messages indefinitely, they pass it to down steam services then forget about it. These are transient systems that dont store data long term. But in a event system events are stored indefinitely, basically like an append only database. Then downstream services can read all of some of these events off the steam as they need to and often watch the streams for new messages to process as they arrive. The big key difference is where the source of truth lives. In message queues the message is passed downstream where downstream is eventually responsible for it. But in event systems the system (like kafka) is responsible for all messages. So downstream dont need to store and data long term (though they often cache it) and can rebuild their caches by rereading the whole event queue again. Which is not possible in a typical message queue system.

(fraud (god tier sre)) #devgang 2021-08-02 18:47
my thoughts RE kafka: kafka is really cool and def the defacto standard for pubsub/messaging queues but it does incur a LOT of operational (and development) complexity. I think w/ RabbitMQ and cloud provider's various pub/sub systems it's now a little more "old school" but seems like a more viable option when you hit a certain throughput (in regards to scale and budget both). However, I've only used RabbitMQ's and aws/gcp/azure's pub sub versions of producer/consumers and not sure if they have streams built in -- which is a big pro of kafka -- it's centralized into a single service rather than parted out into multiple. Nitpick: if you have a Java or language-androgynous stack kafka is a little easier to work into your ecosystem, but if you're sold on something like Go (or specifically a language thats NOT java due to the baggage that comes with it) you kind of have to suck it up and do some java-operations stuff as it pertains to operating a java service. Neither-here-nor-there: Kafka forces you to setup zookeeper which could be pro if you want to centralize your config management, if not, it's just another dependency you need to manage that comes with the system up front What I would do: use a managed kafka service like MSK

Snowie (D4rk) 2021-08-02 20:32
i mean they are all pretty similar - at the end of the day, it is a way to distribute message around. Kafka has some special designs that make it better for event systems, like the replayability factor. ActiveMQ (which is basically SQS) and rabbitmq can effectivly do all the same things, they just have slightly different goals
personally Kafka is my favorite queue system, although rabbitmq is def up there to. I find that rabbitmq starts struggling in larger deployments, or when you are doing streamed data. Kafka because of its pointer byte system, allows you to easily stream large sets of data (terabytes even if you wanted to) to consumers, with really very little problem - which makes it pretty ideal for like video or audio trascoding or the like.
