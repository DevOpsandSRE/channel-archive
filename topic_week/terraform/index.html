#freefraud #devgang 2021-07-27 04:41
a few thoughts of mine that might be beneficial: - terraform workspaces are a godsend when it comes to managing multi environment setups - try not to hardcode input variable values, keep as much of it as you can in a tfvars file - keep groups of dependencies consolidated into subfolders, in order to keep state-files separate - lean heavily on modules, and keep as many resources as you can inside those modules. additionally, version those with a git tag every-time you merge a PR into it, so you can easily roll back a set of resources - keep all your tf (that implements the modules ofc) in a single repo, in order to maintain a single source of truth regarding the active state of your infra. keep modules as independent repos so you can version them seperately
this is based on past exp using terraform at a few different places. "best practice" as far as terraform goes is a bit iffy, as it's not really defined what is or what isnt great to do
some additional tooling that you may find beneficial, although not totally necessary: - https://terragrunt.gruntwork.io/ - https://github.com/gruntwork-io/terratest - https://www.runatlantis.io/

This is what we do. Having terraform pull from git for lots of modules slows things down a lot and makes it hard to keep track of what it using what version. Nous 2021-07-27 17:05
do you do each resource in its own module,
What is the point in having each resource in its own module? Module are for logically grouping thing and terraform does not really handle small module that well IMO as it makes things very hard to refactor. Generally we stick with fewer larger purpose built modules. In terraform being completely DRY is not a good goal to go for as it is very annoying to refactor things (as basically moving or renaming anything requires a state move with is out of band and error prone process).
And I would logically group resources - ServiceA vs ServiceB rather than by resource type.
kryptonite30 2021-07-27 17:08

Nous 2021-07-27 17:08
We used to group things by resource type (don't know why was before I joined) and it becomes a painful dance of complex pipelines and applying thing in the right order... It just causes pain for everyone involved as chaning very often happen across resource types so even simple things end up needing complex deployment processes. Having things grouped by Service or really by how they tend to get modified makes everything a lot simpler.
We have one network module (and a few other shared things like ones for EKS clusters) that outputs things other modules can use.

NOTE: Every Terraform module that provisions an EKS cluster has faced the challenge that access to the cluster is partly controlled by a resource inside the cluster, a ConfigMap called aws-auth. You need to be able to access the cluster through the Kubernetes API to modify the ConfigMap, because there is no AWS API for it. This presents a problem: how do you authenticate to an API endpoint that you have not yet created?
Is definitely the biggest trap when creating a cluster in TF, especially with how the providers work.
And not only that issue - but the aws-auth config map needs to be created before the node groups or else they create a default one helpfully for you causing TF to fail since the resource already exists.
