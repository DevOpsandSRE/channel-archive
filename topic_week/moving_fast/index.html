"According to some studies, in terms of security, more than half of organizations are now running over 41% of their workloads in public clouds, compared to just one quarter in 2019, and this will increase further by the end of 2021. Enterprises are moving fast to the cloud, but they are also finding that things are getting broken during this process. The research showed 11% of organizations reported a cloud security incident in the past year, with the three most common causes being cloud provider issues (26%), security misconfigurations (22%) and attacks such as denial of service exploits (20%). In terms of the business impact of these disruptive cloud outages, 24% said it took up to three hours to restore operations, and for 26% it took over half a day. As a result, it’s no surprise that organizations have significant concerns about enforcing and managing security in the cloud. Their leading concerns were maintaining overall network security, a lack of cloud expertise, problems when migrating workloads to the cloud and insufficient staff to manage their expanded cloud environments. So, what are the root causes of these cloud security concerns and challenges, and how should enterprises address them?" Nous 2021-12-22 10:25
Trunk based development is when all developers commit directly to the main/master/trunk branch (old cvs systems like svn called it trunk, git uses main or master, though it can be called what ever you want). Or to a short lived branch and pull requested into the main branch within a short period of time. In both cases you should be getting your stuff into the main branch at least every day (ideally multiple times a day). The idea is that changes are constantly being added to the main branch so that no one is ever very far behind what is in main and thus what everyone else is working on. This prevents most of those large 500+ line commits which are always very hard to review (and so often just get rubber stamped) as well as all of the conflicts that will arise when lots of people are working on separate things in isolation. Everything in the main branch should be fully tested, and when ever a failure is detected then it should be the top priority to fix it, ideally so that it is never failing for more than 10-30 mins. Feature flags are for when a feature takes longer than a day to complete, as they often do. The basic idea is to put every new feature behind a runtime flag - so that it can be toggled on or off at runtime. This way you can keep committing and pushing to the main branch even when your feature is not finished (as long as the code does not break anything else) and no one is ever far behind the main branch. Then for each environment you have you can toggle the feature on or off depending on if you want it live or not. This allows you to have the same code base in all environments while a different set of features enabled in each one (so you can enable new features in a demo/staging environment first or even to a subset of users in prod).
2
This way if there is an issue with a deployment you can roll back or fix forward the deployment easily (as there wont be many changes in it). And if there is an issue enabling a feature then you can disable it quickly again without needing to do a rollout. And still have the option for keeping it enabled for a small subset of users or environments to debug the issue if required.

Nous 2021-12-22 13:29
Essentially, yeah, a bunch of if-else gates. Possibly with the condition being doing more logic to tell if the feature should be enable or not. But one of the keys is to minimise the number of features that are togglable by ideally removing the flag and condition check once the feature has been fully released. You might want to look at https://launchdarkly.com/ and the docs/guides around it as it is a service entirely dedicated to feature flags (even for web/mobile devices). Also, you can often do the backend work and enable it ahead of time and a frontend change does not always need a new backend feature change. Though you can flag both behind the same or similar flags. IMO I would use a different flag for the backend and frontend - ideally flags should do just one thing so they are easy to remove later on.
