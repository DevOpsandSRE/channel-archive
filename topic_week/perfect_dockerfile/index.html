Let's discuss writing Dockerfiles!
Zig 2021-11-01 05:20
This week's discussion is "Creating Dockerfiles the Right Way". Dockerfiles are the necessary files needed in order to create a Docker image. We see so many different ways they are being written from tutorials to people working on their next project. The real question is, what's the correct way to write them? How do we ensure the image is smallest as possible by thinking about image layers? What is the purpose of a multi-stage Dockerfile? When do we specify "EXPOSE"? When it is necessary to have a multi-line RUN command? When is it best to use CMD over ENTRYPOINT and vice versa? Etc. There is so much to think about. Let's discuss this topic this week with these things in mind as well as other things you may or may not specify in a Dockerfile. Feel free to also share examples. (edited)

If your applications allow it and your code isn’t sensitive to musl’s allocator performance (which it probably isn’t), use Alpine Linux instead of wrestling with aptitude on Ubuntu:
FROM alpine # super fast, leaves no redundant artefacts on the disk RUN apk add --no-cache -q git curl
 (edited)
If you’re compiling from a language that can make standalone executables, consider a multistage build where the last stage copies a statically-linked executable into a scratch image (if you don’t need to do things like make HTTPS requests to the public web, which usually requires the root CA certificate bundles) or the distroless/base distroless/static image (cheers, @Nous, for pointing this one out many times!), which contains the root certificate bundles, to minimize the size and attack surface of your container. (edited)
If you can’t build standalone executables, you might still be able to minimize the size and attack surface by copying the final artefacts into a distroless image instead of running them from the language’s image: https://github.com/GoogleContainerTools/distroless There are images for Java, Node, and (experimentally) Python 3.

Nous 2021-11-03 15:25
You should not use :latest or any moving tag inside kube. You have no idea what version you are running and kube will not automatically upgrade your running containers to the new version. In fact it might not even start new pods on the new version when they are recreated if the image already exists on the node. You can mitigate that latter one by setting the pull policy to always - but then you void the nodes local image cache completely and will always pull the image down even when you dont need to (which can have higher costs or use up more of the rate limit now imposed by dockerhub).
1
IMO versions should be bumped automatically in your repos via something like dependabot. Especially those you use in Dockerfiles. But IMO it is better to use :latest (or at least a moving tag that points to the major version you want) for image building where you dont have this automation set up (and no one will ever manually bump the versions often enough to mitigate security issues. In a build context you should be running tests on things anyway to spot any regressions with either of these. But you really should not leave bumping versions to a human task as it is a boring and very easy to ignore task.

i guess i'll add my thoughts on the topic. i'm going to go a bit further than the syntax or the Dockerfile and pose some things to thing about in general when building container or Docker images. a lot of this stuff comes from general knowledge and use over the last 6+ years: * think about build vs runtime and whether certain parts of the image should be baked in or passed during runtime. i often see environment variables set as default or even secrets accidentally baked in as files. take note of these things as they can bite you later on * weigh image size and build time versus optimization and compatibility. it is easy to chase the Alpine base image rainbow and paint yourself into other compatibility and headaches where your base images don't line up with existing tooling. you may find that image size or build times are not your priorities * automate and document your image build process. a Dockerfile should be considered code and if there are runtime and buildtime requirements without documentation then the next person to come along and try to build or use the image will run into headaches * think about reproduction and drift. will this Dockerfile build the same in a year? does it rely on pulling things from a git main branch that could be very different days, weeks, or months from now?

Nous 2021-11-05 16:53
Distroless is based on debian systems. But it only contains enough from debian to do what it needs to do. It has no package manager or anything not strictly required to run what each one is designed for
So it is not really a valid debian distro. And distroless is really just a name for the containers, not really a technical term for this image has no distro. It is meant to convay that the images do not contain a fully working linux distro as you would typically expect - only the bare minimum required to run some application.
And since distroless/static contains glibc it is important to know which version and where to get the same version. So when building for a distroless container you can use the -debian11 image to know that you should use a builder container that matches that to have a compatible version of glibc. Since that is where they got their version of glibc from.

andyshinn
i guess i'll add my thoughts on the topic. i'm going to go a bit further than the syntax or the Dockerfile and pose some things to thing about in general when building container or Docker images. a lot of this stuff comes from general knowledge and use over the last 6+ years: * think about build vs runtime and whether certain parts of the image should be baked in or passed during runtime. i often see environment variables set as default or even secrets accidentally baked in as files. take note of these things as they can bite you later on * weigh image size and build time versus optimization and compatibility. it is easy to chase the Alpine base image rainbow and paint yourself into other compatibility and headaches where your base images don't line up with existing tooling. you may find that image size or build times are not your priorities * automate and document your image build process. a Dockerfile should be considered code and if there are runtime and buildtime requirements without documentation then the next person to come along and try to build or use the image will run into headaches * think about reproduction and drift. will this Dockerfile build the same in a year? does it rely on pulling things from a git main branch that could be very different days, weeks, or months from now?
