Zig 2021-08-09 17:05
Well the why - and this this is where I've seen it - imagine you having a large cluster of a particular application/service. The nodes in this scenario talk to each other and know about each other. Imagine now that you need to grow your cluster by 25 nodes. You deploy them and they now have their proper configuration to start the service, but now the existing cluster doesn't know these new 25 nodes exist. This is where you need service discovery. An agent is installed on each server (consul, for example) that sends their data to a large consul cluster. This cluster would be used by many applications in your infra that would be "queried" from so that applications and services know what nodes are in the cluster. You can also add tags and whatnot that show "healthy" / "Unhealthy" status for a particular node's app so that the other nodes in the cluster dont sent it traffic until it's in a healthy state (Either a problem or perhaps the node is getting provisioned and not yet ready to serve traffic)

Nous 2021-08-09 19:01
Consul has two parts, servers and agent. The servers form a cluster of nodes that will elect a leader among them and forward requests to it. The agents live on the same nodes as the services they act on and join the cluster. All the nodes in the cluster talk to each other though sharing information about which nodes are alive or dead and relay this to other nodes. This is not a master/slave architecture.
