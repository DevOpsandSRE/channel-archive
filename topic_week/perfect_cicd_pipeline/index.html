This week's topic is "Building the Perfect CI/CD Pipeline"! Let's discuss how one should be built. Does the one you currently use work? What needs to be improved? What suggestions do you have for others when building a Pipeline? "A CI/CD pipeline is a series of steps that must be performed in order to deliver a new version of software. Continuous integration/continuous delivery (CI/CD) pipelines are a practice focused on improving software delivery using either a DevOps or site reliability engineering (SRE) approach. A CI/CD pipeline introduces monitoring and automation to improve the process of application development, particularly at the integration and testing phases, as well as during delivery and deployment. Although it is possible to manually execute each of the steps of a CI/CD pipeline, the true value of CI/CD pipelines is realized through automation. Elements of a CI/CD pipeline The steps that form a CI/CD pipeline are distinct subsets of tasks grouped into what is known as a pipeline stage. Typical pipeline stages include: - Build - The stage where the application is compiled. - Test - The stage where code is tested. Automation here can save both time and effort. - Release - The stage where the application is delivered to the repository. - Deploy - In this stage code is deployed to production. - Validation and compliance - The steps to validate a build are determined by the needs of your organization. Image security scanning tools, like Clair, can ensure the quality of images by comparing them to known vulnerabilities (CVEs)." (edited)

Scratch 2021-09-27 15:46
We have CRs (Code Reviews) which are just a pull request. We have home built systems for everything, git server, viewing code online, performing CRs. When a CR is created a team can choose to have linters run against it. My team doesn't run linters currently, but we've been discussing the possibility of adding them. The CR includes the version set revision (so all of the dependency versions) and a dry-run is performed of the branch which runs the unit tests. Then any other team member can review it. For more important changes it's important that an SME reviews it. We have some policies in place, but no software enforcing that.
Once the CR is merged, that kicks off our deployment pipeline. We have Beta stages which are for us, Gamma stages which are for internal teams, then Prod. Then each of those have a OneBox deployment which is a single unit of the service, but shares a small bit of the load. So it verifies the changes before deploying to the rest of the stage. Then we have waves of deployments of increasing size.

Zig 2021-09-27 16:03
We dont always pin to latest - and when I think of this, I think of like containers. We usually pin to the version number of the CD run that built the image. The reason is, we may push code, but we dont necessarily want to merge it to staging or production right away. We may want to roll the latest version to staging nodes first to see how it will impact the service. Or perhaps, upgrading the version may restart the service and production shouldnt be restarted since that may also cause some downtime. So we have to manually update each node to latest at times in the evening.
example -think of Kafka running in containers. You cant just restart the whole cluster at once. That will lead to under replicated or offline partitions (loss of data) since Kafka works in a clustered fashion. Usually you can restart one or two servers at a time.

Nous 2021-10-02 09:44
Jenkins was a first gen CI system. Everything is configured in the UI, which makes it hard to see/approve changes, things are live straight away so any larger changes can break things while you play around with them to see if they will work (as oppose to testing them in a branch first). They where designed to run agents directly on servers which bloated after a while with every tool ever single pipeline needed. Different versions of packages meant to needed different agents and managing it all is a huge pain. Often java focused and everything else is a second class citizen. This also goes for other older tools like teamcity, stash etc. Jenkins is also plugin hell, everything is a plugin and you need a whole bunch to make it a useable tool these days. Which increases the learning curve and makes the whole thing a mess. I would not use a first gen CI system any more. They should just be left to die. They have tried to tack on more modern features but have all done it badly to keep support for the older ways of doing things. Second gen CI systems are all config first, where the config is in the repo next to the code. As well as Docker (or at least ephemeral VMs for each job) so the agents are kept clean and essentially reset after each job and each job can install the tools it needs for that job. This includes things like travis, circle ci, github actions, gitlab ci, drone ci etc... so many choices there is no reason to use the legacy tools any more. I think there is also a third gen popping up now - more abstract than just CI and starting to go down general workflow pipelines. Often kube focused and while you can build CI/CD systems out of them they are more general and no longer tied to only being triggered by Git event but rather any event. This includes things like Tekton, Argo Wrokflows etc type tools.

Nous
Jenkins was a first gen CI system. Everything is configured in the UI, which makes it hard to see/approve changes, things are live straight away so any larger changes can break things while you play around with them to see if they will work (as oppose to testing them in a branch first). They where designed to run agents directly on servers which bloated after a while with every tool ever single pipeline needed. Different versions of packages meant to needed different agents and managing it all is a huge pain. Often java focused and everything else is a second class citizen. This also goes for other older tools like teamcity, stash etc. Jenkins is also plugin hell, everything is a plugin and you need a whole bunch to make it a useable tool these days. Which increases the learning curve and makes the whole thing a mess. I would not use a first gen CI system any more. They should just be left to die. They have tried to tack on more modern features but have all done it badly to keep support for the older ways of doing things. Second gen CI systems are all config first, where the config is in the repo next to the code. As well as Docker (or at least ephemeral VMs for each job) so the agents are kept clean and essentially reset after each job and each job can install the tools it needs for that job. This includes things like travis, circle ci, github actions, gitlab ci, drone ci etc... so many choices there is no reason to use the legacy tools any more. I think there is also a third gen popping up now - more abstract than just CI and starting to go down general workflow pipelines. Often kube focused and while you can build CI/CD systems out of them they are more general and no longer tied to only being triggered by Git event but rather any event. This includes things like Tekton, Argo Wrokflows etc type tools.
