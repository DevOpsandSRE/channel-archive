Nous 2022-01-10 20:22
The first company I worked for had this PHP extension written in C where all our financial data went though it. It was basically a glorified shared memory access to count requests then in a 15 minute cron it would dump this data to a database. It would deadlock multiple times a day across over some of the ~2000 VMs we were running. When it did the only way to get it running again was to clear out the shared memory - losing all the data in it for that whole machine. The extension was basically a glorified counter using shared memory so would take out the whole VM every time. So up to 15 minutes of good data, plus how ever many minutes of a locked up node it took us to clear it once we noticed. And when I stared load balancing was static, so traffic would still go to the broken node until it was fixed... It took us 4 years to replace it because the dev that wrote it (and only one that understood it) refused to believe there was any better way of doing what it did (despite many attempts at designing a different solution and the constant evidence it was throwing away finical data). He would work on this software live on a production server and then when he wanted to release it he got one of the sysadmins to copy it to all the ~2000 servers and recompile it. It was not in git or any version control system. It also used autoconf which generates a makefile, but he had manually modified the makefile to change a few things which meant you had to modify it again every time you ran ./configure . The worst Dev I ever worked with, stuck in the 1990s and refused to learn anything other than PHP, C, Cron and MySql.

We used to have people do that all the time. One of the worst features of crontab. crontab -e which you use all the time is one finger slip away from -r... and there is no confirmation on it at all. IMO dont use crontabs, use systemd timers where you can. Much better overall. But fi you must use cron, use the cron files in /etc/cron.d and never use crontab.
